# Origin Freshness Index Design

## Summary

RenderMesh needs a per-origin freshness index that records the known state of
all files in an origin source. The index is built by listing the source of data,
capturing file paths and metadata, and storing that inventory in memory. On each
refresh interval, RenderMesh lists the origin again, compares the new inventory
against the previous one, updates only changed files in the local mirror, removes
deleted files, and recompiles Handlebars AST entries only for changed HTML
template candidates.

This gives RenderMesh fine-grained control over freshness without reading the
source data provider during normal request handling.

## Goals

- Maintain one in-memory inventory per origin.
- Track every source object path known for that origin.
- Track source metadata:
  - file path/key
  - creation date when the provider exposes it
  - last modified date when available
  - size
  - provider fingerprint such as ETag when available
  - time when RenderMesh captured the metadata
- Refresh the inventory periodically per origin.
- Detect added, modified, unchanged, and removed files.
- Download/copy only added or modified files.
- Remove local mirror files when source files disappear.
- Recompile Handlebars AST only for modified or added template candidates.
- Remove template AST entries when template files disappear or stop being HTML.
- Keep render-time reads served from the activated local mirror and in-memory
  template registry.

## Non-Goals

- No request-time source reads.
- No distributed freshness coordination between RenderMesh instances.
- No push invalidation API in this iteration.
- No filesystem watcher in the local provider MVP.
- No global cache semantics beyond the origin mirror and template registry.

## Core Model

Each origin should have an in-memory `OriginFreshnessIndex`.

Conceptual shape:

```rust
pub struct OriginFreshnessIndex {
    pub origin_id: String,
    pub captured_at: DateTime<Utc>,
    pub files: BTreeMap<String, OriginFileState>,
}

pub struct OriginFileState {
    pub path: String,
    pub created_at: Option<DateTime<Utc>>,
    pub last_modified: Option<DateTime<Utc>>,
    pub captured_at: DateTime<Utc>,
    pub size: u64,
    pub etag: Option<String>,
    pub content_type: Option<String>,
    pub cache_control: Option<String>,
}
```

The stored `path` is the provider-independent object key used by the mirror, for
example `index.html` or `_rendermesh/edge.yaml`. It must never be absolute and
must never contain parent-directory traversal.

## Provider Contract

Storage providers should expose a listing API that returns summaries. The
existing `RemoteStorage::list_objects` already has most of the required shape,
but it needs to represent creation time separately when the provider has it.

Desired summary fields:

```rust
pub struct RemoteObjectSummary {
    pub key: String,
    pub created_at: Option<DateTime<Utc>>,
    pub last_modified: Option<DateTime<Utc>>,
    pub captured_at: DateTime<Utc>,
    pub size: u64,
    pub etag: Option<String>,
    pub content_type: Option<String>,
    pub cache_control: Option<String>,
}
```

Provider notes:

- S3 usually exposes `last_modified`, `size`, and `etag`, but not object creation
  date. `created_at` should be `None`.
- Local filesystem origins can expose platform creation time when available, but
  should set `created_at` to `None` when unavailable.
- `captured_at` is always generated by RenderMesh at the moment the listing is
  captured.

## Refresh Algorithm

For each origin refresh:

1. List the source provider.
2. Normalize and validate every returned key.
3. Build a new `OriginFreshnessIndex` with `captured_at = now`.
4. Diff the new index against the previous in-memory index for that origin.
5. Classify paths:
   - `added`: path exists only in new index.
   - `modified`: path exists in both, but freshness metadata changed.
   - `unchanged`: path exists in both and freshness metadata matches.
   - `removed`: path exists only in previous index.
6. Fetch only `added` and `modified` files from the provider.
7. Write changed files into a staging mirror.
8. Remove `removed` files from the staging mirror.
9. Compile or update templates for changed template candidates.
10. Remove template entries for removed files or files that are no longer HTML.
11. Activate the new origin snapshot only after mirror changes, edge config
    parsing, and template updates all succeed.
12. Replace the origin freshness index with the new index.

If any required step fails, RenderMesh should keep serving the previous origin
snapshot and previous freshness index.

## Change Detection

A file is unchanged when all available freshness fields match:

- `size`
- `etag`, when the provider supplies it
- `last_modified`, when the provider supplies it
- `content_type`, when the provider supplies it
- `cache_control`, when the provider supplies it

`captured_at` must not make a file modified by itself. It records when the
metadata was observed, not source freshness.

If a provider cannot supply a strong freshness signal, RenderMesh should fall
back to conservative behavior:

- If size changed, modified.
- If last modified changed, modified.
- If ETag changed, modified.
- If metadata is insufficient and content correctness matters, fetch and hash
  the file as a provider-specific fallback.

## Template Candidate Rules

A file is a template candidate when:

- Its normalized key ends with `.html` or `.htm`, or
- Its source/mirror content type is HTML.

On refresh:

- Added HTML candidate: fetch/copy the file, compile it, and insert its AST.
- Modified HTML candidate: fetch/copy the file, recompile it, and replace its
  AST.
- Removed HTML candidate: remove its AST.
- Modified non-HTML file: do not compile.
- File changed from HTML to non-HTML: remove its old AST.
- File changed from non-HTML to HTML: compile and insert its AST.

The AST registry should be per origin. A failure to compile any changed template
for an origin should prevent activation of that origin's new snapshot. The
previous template registry should remain active.

## Origin Snapshot

The freshness index should feed a coherent per-origin snapshot.

Conceptual shape:

```rust
pub struct OriginSnapshot {
    pub origin_id: String,
    pub generation: u64,
    pub index: Arc<OriginFreshnessIndex>,
    pub mirror_root: PathBuf,
    pub edge_config: Arc<EdgeConfig>,
    pub templates: Arc<Handlebars<'static>>,
    pub activated_at: DateTime<Utc>,
}
```

Request handling should use the latest activated snapshot for the resolved
origin. It should never observe a mirror from one generation with templates from
another generation.

This is the key improvement over the current behavior, where the mirror is
activated first and templates are refreshed afterward.

## Memory Ownership

All per-origin runtime state should be owned by a main runtime state object and
shared through `Arc`:

- origin freshness index
- active origin snapshot
- edge config
- Handlebars registry

When a refresh succeeds, RenderMesh builds a new snapshot and swaps the active
`Arc` for that origin. In-flight requests can finish using the previous snapshot
while new requests use the new snapshot.

## Interaction With Local Origins

This design applies to both S3 and local origins:

```text
S3 list objects      -> OriginFreshnessIndex -> diff -> mirror/template update
Local directory walk -> OriginFreshnessIndex -> diff -> mirror/template update
```

The provider only changes how listing metadata and file bodies are retrieved.
The diff, mirror update, template update, and snapshot activation logic should be
provider-independent.

## Current Gap

The current implementation already does parts of this:

- It lists remote objects during sync.
- It compares local metadata to remote summaries.
- It downloads changed files.
- It removes deleted files.
- It reloads the whole origin template registry after sync.

The missing pieces are:

- No explicit in-memory origin freshness index.
- No persisted or in-memory `captured_at` for listings.
- No creation time field.
- No diff object that classifies added/modified/removed/unchanged paths.
- No incremental template AST update.
- No atomic origin snapshot that ties mirror, edge config, freshness index, and
  templates to the same generation.

## Testing Plan

Unit tests:

- Builds an index from provider summaries.
- Records `captured_at` for each listed file.
- Treats changed `captured_at` alone as unchanged.
- Detects added files.
- Detects modified files by size.
- Detects modified files by ETag.
- Detects modified files by last modified date.
- Detects removed files.
- Rejects invalid provider keys.
- Classifies HTML template candidates.
- Removes AST when an HTML file is removed.
- Removes AST when an HTML file becomes non-HTML.
- Recompiles AST when an HTML file changes.

Service tests:

- Refresh fetches only added and modified files.
- Refresh does not fetch unchanged files.
- Refresh keeps previous snapshot if template compilation fails.
- Refresh keeps previous snapshot if edge config parsing fails.
- Refresh swaps snapshot after mirror, edge config, and templates all succeed.
- In-flight snapshot access remains valid after a newer snapshot activates.

Integration tests:

- A changed HTML file becomes visible after refresh.
- A changed Handlebars template renders with the new AST after refresh.
- A deleted template no longer renders after refresh.
- A failed template update keeps serving the previous rendered output.

## Acceptance Criteria

- Each origin has an in-memory inventory of source file freshness metadata.
- Refresh is per origin and interval-based.
- Only changed source files are fetched or copied.
- Template AST updates happen only for changed template candidates.
- Deleted templates are removed from the active template registry after a
  successful refresh.
- Mirror files, edge config, freshness index, and templates activate as one
  generation.
- Failed refreshes keep the previous generation active.
